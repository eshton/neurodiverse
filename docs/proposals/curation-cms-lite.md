# Proposal: git-backed CMS for non-technical curators (the least-setup option)

**Status:** recommended path for adding collaborative curation · **Companion doc:** [curation-full-backend.md](curation-full-backend.md) (the heavyweight alternative)

Goal: let non-technical people curate the data through a friendly web dashboard **without changing the architecture**. A git-backed CMS gives editors form-based editing and a "publish" button, but behind the scenes it commits markdown to this repo (or opens a PR). The site stays static, git stays the datastore, and everything valuable about the current setup is preserved.

## Why this fits (what it preserves)

Because edits still land as git commits/PRs, you keep all of it for free:

- **Review/approval, history, attribution, rollback** — native to git.
- **Validation** — the CI already runs the Zod schema + `content.json` drift check on every push/PR, so a malformed entry can't merge.
- **The open-source `content.json`** — regenerated by `prebuild` on each build, unchanged.
- **Privacy, $0 static hosting, SEO** — untouched; no database, no server storing user data.
- **The agentic-curation roadmap** — agents opening PRs and humans editing via the CMS both commit to git, so they compose.

The only genuinely new piece (and only for the self-hosted variant) is a tiny GitHub OAuth shim — not a database, not an SPA.

## Two ways to run it

### Option A — Pages CMS (hosted, least setup) ← lowest effort

[pagescms.org](https://pagescms.org) — a hosted, open-source, GitHub-backed CMS. Editors sign in with GitHub; you install a GitHub App on the repo; a `.pages.yml` at the repo root defines the collections. Zero self-hosted infrastructure, no OAuth function to run.

**Setup (about an hour):**

1. Go to pagescms.org, sign in with GitHub, install the GitHub App on `eshton/neurodiverse` (can be scoped to just this repo).
2. Add `.pages.yml` at the repo root (config sketch below).
3. Invite curators — they get a web dashboard with a form per category; saving commits to the repo (configure a branch + PR if you want review before publish).
4. Point media at `public/covers` so image uploads land where covers live.

**Trade-off:** relies on a third-party hosted service for the editing UI (the *site* and *data* remain yours in git; only the editing dashboard is hosted). Editors need GitHub accounts (free; the App handles authorization).

### Option B — Sveltia CMS (self-hosted UI, still light)

[Sveltia CMS](https://github.com/sveltia/sveltia-cms) is a modern, drop-in replacement for Decap/Netlify CMS. The editing UI is a single static page you host yourself at `public/admin/`, configured by a `config.yml` (Netlify-CMS format). It commits to the repo via the GitHub API.

**Setup (about half a day):**

1. Add `public/admin/index.html` that loads the Sveltia CMS script, plus `public/admin/config.yml` (Netlify-CMS collection format — same field mapping as below, different syntax).
2. Stand up a **GitHub OAuth handshake** — a small Cloudflare Pages Function (the one bit of "backend"; ~30 lines) or a hosted OAuth relay. This is what lets editors log in with GitHub from your own domain.
3. Configure `media_folder: public/covers`, `public_folder: /covers`, and optionally `publish_mode: editorial_workflow` for PR-based review.

**Trade-off:** you host the admin page and run the OAuth function, but nothing else changes. Fully self-owned, no third-party editing host.

> Decap CMS is the older, still-supported sibling of Sveltia with the same config format — mention only if Sveltia doesn't fit.

## Config sketch (illustrative — mirror `content.config.ts`)

Both options need a config that mirrors the Zod schema in `src/content.config.ts`. Below is a representative `.pages.yml` (Pages CMS syntax) for **two** categories — the base fields plus one located category (`diagnosis`) and one web category (`books`). The other 11 categories follow the same pattern; `content.config.ts` is the source of truth to mirror, and the Hungarian field labels already exist in `src/lib/categories.ts` (`FIELD_LABELS`) to reuse.

```yaml
# .pages.yml  — verify exact field syntax against current Pages CMS docs at setup
media:
  input: public/covers      # uploads land here
  output: /covers           # referenced as /covers/... in coverImage

content:
  - name: diagnosis
    label: Diagnózis
    type: collection
    path: src/content/diagnosis/hu
    filename: '{fields.slug}.md'      # or let the editor name the file
    fields:
      - { name: title,        label: Cím,                 type: string,  required: true }
      - { name: topics,       label: Témák,               type: select,  options: { values: [adhd, autism], multiple: true }, required: true }
      - { name: summary,      label: Összefoglaló,        type: text,    required: true }
      - { name: providerType, label: Típus,               type: select,  options: { values: [doctor, clinic, foundation, hospital] }, required: true }
      - { name: city,         label: Város,               type: string,  required: true }
      - { name: address,      label: Cím,                 type: string }
      - { name: lat,          label: Szélesség (lat),     type: number }
      - { name: lng,          label: Hosszúság (lng),     type: number }
      - { name: contact,      label: Elérhetőség,         type: string }
      - { name: priceNotes,   label: Ár megjegyzés,       type: string }
      - { name: sourceUrl,    label: Forrás URL,          type: string,  required: true }
      - { name: coverImage,   label: Borítókép,           type: image }
      - { name: ageGroup,     label: Korosztály,          type: select,  options: { values: [children, adults, both] }, default: both }
      - { name: availability, label: Elérhetőség,         type: select,  options: { values: [in-stock, unavailable] }, default: in-stock }
      - { name: tags,         label: Címkék,              type: string,  list: true }
      - { name: locale,       label: Locale,              type: select,  options: { values: [hu, uk, us] }, default: hu, required: true }
      - { name: lastReviewed, label: Utoljára ellenőrizve, type: date,   required: true }
      - { name: curatedBy,    label: Kuráció,             type: select,  options: { values: [manual, agent] }, default: manual }
      - { name: body,         label: Leírás,              type: rich-text }

  - name: books
    label: Könyvek
    type: collection
    path: src/content/books/hu
    filename: '{fields.slug}.md'
    fields:
      - { name: title,   label: Cím,          type: string, required: true }
      - { name: topics,  label: Témák,        type: select, options: { values: [adhd, autism], multiple: true }, required: true }
      - { name: summary, label: Összefoglaló, type: text,   required: true }
      - { name: author,  label: Szerző,       type: string, required: true }
      - { name: isbn,    label: ISBN,         type: string }
      - { name: sourceUrl, label: Forrás URL, type: string, required: true }
      - { name: coverImage, label: Borítókép, type: image }
      # ...same base tail: topics/tags/availability/ageGroup/locale/lastReviewed/curatedBy/body
```

## Things to get right (gotchas)

- **The config mirrors the schema — keep them in sync.** The CMS config duplicates `content.config.ts` (widgets per field). If you add a schema field, add it to the CMS config too, or editors can't set it. (Same class of drift the `FIELD_LABELS` consolidation addressed — there's no way to auto-derive a CMS config from Zod today.)
- **`ratings` is an array of objects** (`source`, `score`, `scale`, `url?`, `note?`, `checkedAt`). Model it as a list-of-objects field. Most entries have `[]` — fine to leave empty.
- **`articles.publishedDate` is a quoted string, not a date.** Configure it as a plain string field (format `YYYY-MM-DD`), *not* a date widget — the schema deliberately uses `z.string()` so it renders as-is (see architecture.md). A date widget would emit an unquoted date and fail Zod.
- **Enums must be `select` widgets** with the exact schema values (`topics`, `kind`, `format`, `providerType`, `publicationType`, `language`, `availability`, `ageGroup`, `entryType`, `curatedBy`) — free-text would fail validation.
- **Geocoding stays manual.** The CMS won't turn an address into `lat`/`lng`. Editors either paste coordinates or you keep the existing Nominatim step. (A map-picker widget can help where the CMS supports one.)
- **Prefer PR/editorial-workflow mode** so contributions are reviewed (and CI-validated) before publish, rather than committing straight to `main`.
- **Don't hand-edit `public/db/content.json`** — it's regenerated on build; the CMS edits the markdown, the build regenerates the JSON, CI checks it matches.
- **Cover images:** the CMS uploads to `public/covers` and commits them. Note the existing guidance in `docs/gotchas.md` about white-on-transparent logos going invisible on the gray marker backdrop.

## Effort

- **Pages CMS (Option A):** ~1 hour to a working dashboard once the collection config is written; the bulk of the work is transcribing all 13 categories' fields into `.pages.yml`.
- **Sveltia (Option B):** ~half a day (admin page + `config.yml` + the OAuth function).

Either way: no rewrite, no database, no change to how the site is built or served — just an editing UI over the repo you already have.
