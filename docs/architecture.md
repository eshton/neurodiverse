# Architecture

## Stack

Astro 7, static output. Tailwind v4 (CSS-first config, `@theme` tokens in `src/styles/global.css`, `@custom-variant dark` for class-based dark mode toggled via `localStorage` + a button in the layout). Content lives as markdown files under `src/content/<category>/<locale>/*.md`, typed via Astro content collections (`src/content.config.ts`). No database — git is the datastore, which is what makes both manual curation and future agentic curation (an agent opens a PR adding/editing a markdown file) work the same way.

Map feature uses Google Maps JS API (see "Why Google Maps, and what it costs" below for why, and what that requires). Geocoding for lat/lng is done once at content-authoring time via the free Nominatim API (see curation-methodology.md), not at runtime — switching the map *display* to Google didn't require re-geocoding anything.

## Client-side data layer & navigation

Two pieces landed together to make the site feel app-like without adding a backend (full rationale in [proposals/client-data-layer.md](proposals/client-data-layer.md)):

- **Committed dataset at `public/db/content.json`** — the whole content set (every category × locale, frontmatter + markdown body) serialized to one JSON file, served at the URL **`/db/content.json`** via Astro's `public/` passthrough. It's ~180 KB raw / ~37 KB gzipped for ~147 items *including bodies*, so the entire dataset can ship to the client — no two-tier index/body split is needed yet. Because it's committed (not just a `dist/` artifact), it doubles as a linkable open-source database.
  - Generated by `scripts/build-db.mjs` (plain Node + the `yaml` dep) which walks `src/content/<category>/<locale>/*.md`, parses frontmatter, applies the same defaults as the Zod schema, and writes deterministic (sorted, no timestamps) JSON so re-running only changes the file when content actually changed. **The schema defaults are duplicated by hand in that script** — `content.config.ts` imports astro-only modules and can't be `require`d from plain Node, so if you add a defaulted field to the schema, add it to `BASE_DEFAULTS`/`CATEGORY_DEFAULTS` there too or the JSON will omit it.
  - Wired as **`prebuild` and `predev`** in `package.json` (plus a manual `npm run build:db`), so `npm run build`/`npm run dev` always regenerate it first and the committed copy can't silently drift from the markdown. The build then copies it into `dist/db/content.json` like any other `public/` file.
- **View Transitions + prefetch** — `<ClientRouter/>` (from `astro:transitions`) is in the `<head>` of *both* layouts, and `astro.config.mjs` sets `prefetch: { prefetchAll: true, defaultStrategy: 'hover' }`. Together these give SPA-style, no-full-reload navigation (links prefetched on hover) over the still-fully-static build — every page remains real pre-rendered HTML for SEO/direct links. The per-item detail pages are unchanged; only navigation between them got smoother. The slide-over-panel-from-dataset idea (rendering detail without navigating at all) is a documented *future* step, not built yet.

## Content model

`src/content.config.ts` defines a `base` Zod schema shared by every category, then extends it per category with category-specific fields via `collectionFor(category, extraFields)`.

### Base fields (every item, every category)

| Field | Type | Notes |
|---|---|---|
| `title` | string | |
| `locale` | `hu \| uk \| us` | Only `hu` has content. Locale is a data dimension, not Astro's built-in i18n routing — adding UK/US later is just adding content files with `locale: uk`/`us` and activating those routes, no schema change. |
| `topics` | array of `adhd \| autism` | min 1. Items can be tagged both. |
| `summary` | string | |
| `tags` | string array | free-form, default `[]` |
| `sourceUrl` | URL | canonical link to the actual thing (purchase page, show page, video, clinic homepage) |
| `coverImage` | string, optional | path under `/covers/`, e.g. `/covers/szetszortsag.jpg`. Self-hosted, not hotlinked — see curation-methodology.md |
| `ratings` | array of `{source, score, scale, url?, note?, checkedAt}` | generic — no aggregation/weighting logic exists yet. Empty array is fine and expected when no real rating exists for that content type (see below). |
| `availability` | `in-stock \| unavailable`, default `in-stock` | for retail-style items (books) going out of print/stock. Don't repurpose this for clinic intake status — use `priceNotes` free text instead (see diagnosis). |
| `entryType` | `full \| episode`, default `full` | `episode` = a one-off piece of content from a channel/show that isn't itself dedicated to ADHD/autism (e.g. one Partizán Podcast episode). `full` = the whole show/channel/book/clinic is the entity. |
| `parentTitle` | string, optional | required alongside `entryType: episode` — the name of the parent show/channel, rendered as a badge on both listing and detail pages. |
| `curatedBy` | `manual \| agent`, default `manual` | not yet used by any actual agent pipeline, but the field exists for when one lands. |
| `lastReviewed` | date | when the entry's data was last checked against its source. Always set this to the date you actually verified it, not the file-creation date. |
| `ageGroup` | `children \| adults \| both`, default `both` | moved here from `development`'s category-specific fields when the map-first homepage needed to filter by age across every pin-able category, not just one. Backfilled with real (non-default) values on schools (all → `children`), diagnosis, and equipment by reading each entry's own `summary`/`tags` — no new research needed, just inferring from what was already written. `communities` now carries real per-item values too (adult clubs/forums → `adults`, parent groups → `children`, umbrella orgs → `both`) since it was seeded with real content. |

### Category-specific fields

- **books**: `author`, `isbn?`
- **podcasts**: `creator`, `platform`, `episodeCount?`
- **videos**: `creator`, `platform`, `viewCount?`, `subscriberCount?` — YouTube has no public star ratings anymore, so view/subscriber counts are the honest substitute; they live here, not in `ratings`.
- **influencers**: `handle`, `platforms` (array, min 1)
- **equipment**: `priceRange?`, `retailerUrls` (array, default `[]`), plus optional `city?`/`address?`/`lat?`/`lng?` — added for the map-first homepage; almost every retailer sourced this session is online-only (no visitable location), so only one entry (`calmer-fuldugo.md` / profi-fuldugó.hu, Esztergom) actually has these populated. Don't expect most equipment entries to ever have a location — that's the honest state of Hungarian sensory-equipment retail, not a gap to rush-fill.
- **food**: `kind` (`recipe \| product`), `dietaryTags` (default `[]`)
- **communities**: `providerType` (`group-therapy \| therapist \| institution \| online-group`), `city?`, `address?`, `lat?`, `lng?`, `contact?` — peer-support groups, self-advocacy associations, community centers, and online communities (forums/Facebook groups). `online-group` is the type for online-only communities (Facebook groups, `asperger.hu`'s forum); those have **no** `city`/`lat`/`lng` (which is why `city` is optional here, unlike diagnosis/schools where it's required) and only ever appear in the `/hu/communities/` listing, never on a map. Physical entries (an association's office, a community center) carry the same geo shape as diagnosis (`city`/`address?`/`lat?`/`lng?`) and do appear on the map. `lat`/`lng`/`address` were added to this schema when the category was first seeded with real content — the map-first homepage's `index.astro` had always *tried* to read `lat`/`lng` off communities, but the fields didn't exist until then, so no community could ever pin before. Facebook-group entries can't have their live member count fetch-verified (login wall), so member counts come from public search data and that limitation is stated in the entry body — see curation-methodology.md.
- **diagnosis**: `providerType` (`doctor \| clinic \| foundation \| hospital`), `city`, `address?`, `lat?`, `lng?`, `contact?`, `priceNotes?`
- **schools**: same shape as diagnosis, `providerType` is `mainstream-integration \| special-needs-school \| kindergarten` instead.
- **development**: therapists/psychologists/centers doing ongoing developmental/therapeutic work (ABA, sensory integration, skills groups, gyógypedagógia) — distinct from `diagnosis` (getting diagnosed) and `communities` (peer support groups/institutions). Same geo shape as diagnosis/schools (`city`, `address?`, `lat?`, `lng?`, `contact?`, `priceNotes?`) plus `providerType` (`psychologist \| therapist \| developmental-pedagogue \| occupational-therapist \| center`) — most real entries are multi-practitioner centers (`center`), individual solo practitioners are the minority. `ageGroup` used to live here specifically; it's on `base` now (see above). The same institution can legitimately have one entry here and a separate one in `diagnosis` if it runs both services (e.g. Autizmus Alapítvány: `autizmus-alapitvany-ambulancia` in diagnosis, `terc` here) — that's not a duplicate, it's two different services at the same address.
- **movies**: `kind` (`documentary \| narrative`), `director?`, `year?`, `runtimeMinutes?`, `streamingUrls` (array, default `[]`) — films/documentaries about ADHD/autism, or with a well-known ADHD/autistic protagonist. Has no geo fields, unlike equipment; it belongs on the `/web/` grid page, never the map. `availability` (a base field) is reused here for "watchable in Hungary right now" (`in-stock`) vs. "no legal HU streaming option found" (`unavailable`) rather than adding a movies-specific field — see content-status.md for which entries landed which way and why.

`CATEGORIES` (exported from `content.config.ts`) is the single source of truth for which categories exist — page routing, homepage cards, and `getStaticPaths` all derive from it. Adding a category = add it to `collections` in `content.config.ts`, add its label (and emoji, for the homepage) to the `CATEGORY_LABELS`/`CATEGORY_META` maps in the three page files, create `src/content/<category>/hu/`.

## Routing

- `src/pages/index.astro` — **the map-first homepage**, `/`. Not under `[locale]` — this site is Hungary-only for the foreseeable future (other countries get their own separate domains, not `/uk/`/`/us/` subpaths on this one), so the new front door doesn't need the locale param. Pulls every geo-tagged item across `diagnosis`/`schools`/`development`/`communities`/`equipment`, hands them to `<MapExplorer>` inside `<MapLayout>`. Used to be a meta-refresh redirect to `/hu/`; `public/_redirects` no longer has a rule for `/` since it's real content now, not a redirect.
- `src/pages/web.astro` — the **grid view**, `/web/`, for content with no physical location (`podcasts`, `books`, `equipment`, `videos`, `movies`). Same header/layout as the map (`MapLayout`), same age-filter behavior, but the "rest of the screen" is a scrollable card grid (`WebExplorer`) instead of a map, and there's no location selector since there's nothing to recenter. Linked from `SiteHeader`'s nav as "Web".
- `src/pages/rolunk.astro`, `adatvedelem.astro`, `gyik.astro` — About/Privacy/FAQ pages linked from `SiteHeader`'s nav, real Hungarian copy (not placeholder). `adatvedelem.astro` documents actual data practices honestly (no accounts/analytics/cookies; geolocation is client-side-only, never sent to a server) and marks the operator's legal identifiers (cégjegyzékszám, adószám, székhely) as **not yet published** rather than inventing them — "Armin & Felix Kft." doesn't turn up in the Hungarian company registry (Nemzeti Cégtár) under a search for those details, so don't treat that name as a verified registered entity without the user supplying real registration data.
- `src/pages/[locale]/index.astro` — the **original** category-grid homepage, still fully functional at `/hu/`, just no longer the default landing page. Kept intentionally, per the user's "keep the current front page for reference or later" instruction — don't delete or "clean up" this route without checking first.
- `src/pages/[locale]/[category]/index.astro` — listing. Topic filter via `?topic=adhd|autism` query param (plain links, no JS). Renders `<ClinicMap>` above the list when the category is in `MAP_CATEGORIES` (currently `diagnosis`, `schools`, `development`, `communities`) and at least one filtered item has `lat`/`lng`. The "back" link at the top routes to `/web/` when the category is in `WEB_CATEGORIES` (`books`, `podcasts`, `videos`, `equipment`, `movies`) and to `/` (the map) otherwise — so content always links back to whichever browsing surface it's actually reachable from, not a hardcoded default.
- `src/pages/[locale]/[category]/[slug].astro` — detail page. `getStaticPaths` walks every category's collection and derives `{locale, category, slug}` from the content file's `id` (glob loader IDs are `<locale>/<filename-without-extension>`, so `slug = id.split('/').slice(1).join('/')`). Renders every extra (non-base) field generically via a label lookup map (`FIELD_LABELS`) — adding a new category-specific field to the schema means adding its Hungarian label there too, or it'll render with the raw key name. Same `WEB_CATEGORIES` back-link routing as the listing page above.

## Components

- `src/components/SiteHeader.astro` — the top nav, shared by both layouts below so the whole site has one consistent header instead of two divergent ones. Brand "Neurodiverz Térkép" (always links to `/`, the map — not to whatever page you're currently on), with the public-domain Neurodivergent Pride Flag (`/neurodivergent-flag.png`, sourced from Wikimedia Commons, public domain — see below) as a small logo next to the wordmark → Web/Rólunk/Adatvédelem/GYIK links → circular profile-picture placeholder with a pure-CSS hover tooltip reading "Hamarosan" (no accounts system exists yet, purely a visual placeholder). **No dark-mode toggle** — removed on request; each page still opens in dark mode automatically if the OS is set to dark (`prefers-color-scheme` check in each layout's `<head>`), there's just no manual override control anymore. If a toggle ever comes back, add it here once, not per-layout.
- **Favicon**: `public/favicon.png`, generated from the same flag image (`sips -p ... --padColor FFFFFF` square-pad + `-Z 64` downscale) rather than an unrelated icon, so the browser tab matches the header logo. Note: a transparent-background version was tried and reverted — the file itself was correctly transparent (verified pixel-by-pixel), but browsers cache favicons independently of normal page cache and the difference is invisible against a white tab bar in light mode anyway, so it wasn't worth the added complexity. If revisiting, test in a dark browser theme, not just a hard-refresh.
- `src/layouts/Layout.astro` — `SiteHeader` + `max-w-4xl` centered scrollable container + footer. Used by every `/hu/...` page and the About/Privacy/FAQ stubs.
- `src/layouts/MapLayout.astro` — `SiteHeader` + full-bleed body (no max-width wrapper, `h-screen flex flex-col overflow-hidden` so the map fills the viewport instead of the page scrolling). Used only by the new `/` homepage — the filter bar is not part of this layout, it's rendered by `MapExplorer` itself as a second row below the header (see below), so this layout stays reusable for any future full-bleed page that doesn't need filters.
- `src/lib/googleMaps.ts` — shared client-side module: `loadGoogleMaps()` (injects the Maps JS `<script>` tag at most once per page, memoized via a module-level promise so multiple map components on the same page await the same load instead of re-injecting), `circularIcon(url, category?)` (canvas-baked circular photo marker, described below), `categoryIcon(category)` (synchronous fallback for items with no `coverImage` — a solid brand-colored circle with the category's emoji, replacing Google's default red teardrop pin, which carries no type information), `getGoogle()`, `CATEGORY_ICONS` (the emoji per category — kept identical to `CATEGORY_META`'s icons on the category-grid homepage on purpose, so "school" means the same glyph everywhere), `MAP_STYLES` (a `styles` array that hides Google's own default POI/transit markers and labels, since they're irrelevant clutter on a directory that has its own pins), `POPUP_BTN_STYLE` (the "open in new tab" popup button, styled inline since `InfoWindow` content is a raw HTML string, not a component), and the `MARKER_SIZE` (`30`)/`BACKDROP`/`POPUP_IMG_STYLE` constants. Both `ClinicMap.astro` and `MapExplorer.astro` import from here — extracted specifically so the full-page map and the embedded per-category maps can't drift out of sync on marker styling.
- `src/components/ClinicMap.astro` — **Google Maps JS API** wrapper (switched from Leaflet/OpenStreetMap; see below for why and what that costs). Takes `clinics: Array<{title, lat, lng, city, address?, href, coverImage?}>`, a `region: 'budapest' | 'hungary'` prop (default `'hungary'`), and an optional `category` prop (applies to every marker in that instance, since a `ClinicMap` on a category listing/detail page only ever shows one category at a time — unlike `MapExplorer`, which needs it per-item). Each region has its own `restriction.latLngBounds` + `strictBounds: true` + `minZoom` so the map can't be panned/zoomed out past that region — Budapest's box is a tight crop around the city, Hungary's covers the whole country. The listing page (`[category]/index.astro`) renders **two separate map instances** for map-enabled categories, splitting the geo-tagged items on `city === 'Budapest'` — one Budapest-only map, one for everywhere else. Detail pages pick the matching region automatically from the single item's `city`. The component supports multiple instances per page via `querySelectorAll('.clinic-map')`, not a single hardcoded `id`. `href: ''` renders the title as plain bold text instead of a link (used for the single-marker map on a detail page, where linking to itself is pointless). Single marker → centered at zoom 14 (clamped to the region's `minZoom`); multiple → `fitBounds`. Clicking the map closes the open `InfoWindow`, same as `MapExplorer`. Every marker is a 30×30 circle (`MARKER_SIZE` in `googleMaps.ts`) with a dark border (`#27272a`, not white — chosen specifically so pins read clearly against any map style instead of blending in) baked via canvas: items with `coverImage` get their photo cropped into the circle (`circularIcon`) plus a small corner badge stamped with the category's emoji so the type still reads at a glance even with a photo showing; items without one get the plain category-colored+emoji circle (`categoryIcon`) instead of Google's default pin. The canvas step also means every marker renders at identical dimensions regardless of the source image's format/aspect ratio. The backdrop baked in behind a transparent source image is neutral gray (`#d4d4d8`) — several scraped logos are white-shapes-on-transparent (styled to sit on a colored header on their source site) and go invisible on a pure white circle; gray gives them contrast without affecting photos, which fill the frame and never show the backdrop anyway. A pure-white-only logo with no other distinguishing color (happened once — adhd-kozpont) is still unusable against *any* flat backdrop; when that happens, drop the image rather than keep a blank-looking marker. Popups use `google.maps.InfoWindow`, opened on marker click, with a plain-text title and an explicit "Megnyitás új lapon ↗" button rather than a same-tab link — see `MapExplorer` below for why.
- `src/components/MapExplorer.astro` — the full-page map on `/`. Unlike `ClinicMap`'s per-listing region lock, this is the primary browsing surface: freely zoomable/pannable, only loosely bounded to Hungary (`strictBounds: false`) so an imprecise geolocation fix near a border doesn't visibly clip. Renders its own filter bar (a second row below `SiteHeader` — cramming the header content *and* a location-select *and* 5 type chips *and* 3 age chips into one literal row wasn't usable, so this ended up as two stacked rows instead of the one row originally described; flagged to the user as a deviation, not a silent one) with: a location `<select>` (Közelben → geolocation, then Budapest, then the 19 counties from `src/data/counties.ts`), 5 type-filter toggle chips (multi-select, default all active — Klinikák/Iskolák/Terapeuták/Közösségek/Boltok mapping onto `diagnosis`/`schools`/`development`/`communities`/`equipment`), and 3 age-filter toggle buttons (single-select — right-aligned with a border-left divider so they read as a visually distinct filter group from the type chips, "Minden korosztály"/Gyerek/Felnőtt — renamed from the original "Mind" specifically because it read as "load all content types", not "any age"). Markers are keyed by `item.href` in a `Map<string, marker>` (`markersByKey`) so a filter change only removes markers that dropped out and adds ones that newly qualify — it does **not** clear and rebuild the whole marker set on every change, which is what caused a visible blink on every selector interaction before this. Newly-added markers (first load, or freshly revealed by a filter/location change) get `google.maps.Animation.BOUNCE` for 1.6s then `setAnimation(null)`, so new pins are easy to spot instead of just silently appearing. Picking a filter doesn't move the viewport, only the location control does — otherwise the map would jump around unexpectedly every time a chip is toggled. "Közelben" calls `navigator.geolocation.getCurrentPosition()` and shows an inline status message on denial/error rather than failing silently. Clicking anywhere on the map itself closes the open `InfoWindow` (Google Maps doesn't do this by default). The popup no longer makes the title a same-tab link — it's plain bold text plus an explicit "Megnyitás új lapon ↗" button (`target="_blank"`), so opening an entry never tears down the map's live state (filters, location, zoom) — this was also the real fix for a reported "header blink on click," which turned out to be the ordinary full-page-navigation flash from leaving the map in the same tab, not a header bug. Type/age filter selections persist across visits via `localStorage` (`neuro-map-filters` key, `{types, age}` shape) — restored on load and re-saved on every change, wrapped in try/catch so a corrupt or unavailable `localStorage` just falls back to the defaults instead of breaking the page. `MARKER_SIZE` is `30` (down from an initial `40` — smaller circles by request). Google's own default POI markers (restaurants, shops, transit stops) are suppressed via a `styles` array (`MAP_STYLES` in `googleMaps.ts`, also applied to `ClinicMap`) so only this site's own pins ever show.
- `src/components/WebExplorer.astro` — the card-grid counterpart to `MapExplorer` for `/web/`. Same filter-bar visual language (type chips + right-aligned age toggle with a divider), minus the location `<select>`. Cards are plain anchor tags (`target="_blank"`, consistent with the map popups' "open in new tab" behavior) laid out in a CSS grid (`repeat(auto-fill, minmax(220px, 1fr))`) inside a scrollable `overflow-y-auto` container below the fixed filter bar. Filtering toggles a `hidden` class on each card client-side (no marker rebuild needed since there's no map) and shows an empty-state message when a filter combination matches nothing. Same `localStorage` persistence pattern as `MapExplorer`, under its own `neuro-web-filters` key so the two pages' filter states don't collide or overwrite each other.

### Why Google Maps, and what it costs

This was Leaflet + OpenStreetMap originally (free, no API key, no billing account, ever) — see `docs/gotchas.md` for Leaflet-specific traps that no longer apply now that it's gone. Switched to Google Maps JS API on request. Requires: a **Maps JavaScript API** key from a Google Cloud project with **billing enabled** (mandatory even to stay inside the free tier — no billing account = the API won't serve requests at all), and that key **restricted by HTTP referrer** to this site's domain(s) so it can't be lifted and abused elsewhere. Pricing as of this write-up: first 10,000 map loads/month free, then $7/1,000 loads — a low-traffic directory site should realistically stay at $0, but there's no "always free, zero account" tier for a real embedded map with custom markers (the genuinely-free options — the legacy `output=embed` iframe, or the Maps Demo Key — don't support custom markers/popups or aren't meant for production, respectively). The key lives in `.env` as `PUBLIC_GOOGLE_MAPS_API_KEY` (the `PUBLIC_` prefix is required for Astro/Vite to expose it to browser code — this is fine because a Maps JS key is meant to run client-side and is secured by domain restriction, not by being secret) and must also be set in Cloudflare Pages' project environment variables for production builds, not just locally — see `docs/deployment.md`.
