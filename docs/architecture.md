# Architecture

## Stack

Astro 7, static output. Tailwind v4 (CSS-first config, `@theme` tokens in `src/styles/global.css`, `@custom-variant dark` for class-based dark mode toggled via `localStorage` + a button in the layout). Content lives as markdown files under `src/content/<category>/<locale>/*.md`, typed via Astro content collections (`src/content.config.ts`). No database — git is the datastore, which is what makes both manual curation and future agentic curation (an agent opens a PR adding/editing a markdown file) work the same way.

Map feature uses Google Maps JS API (see "Why Google Maps, and what it costs" below for why, and what that requires). Geocoding for lat/lng is done once at content-authoring time via the free Nominatim API (see curation-methodology.md), not at runtime — switching the map *display* to Google didn't require re-geocoding anything.

## Client-side data layer & navigation

Two pieces landed together to make the site feel app-like without adding a backend (full rationale in [proposals/client-data-layer.md](proposals/client-data-layer.md)):

- **Committed dataset at `public/db/content.json`** — the whole content set (every category × locale, frontmatter + markdown body) serialized to one JSON file, served at the URL **`/db/content.json`** via Astro's `public/` passthrough. It's ~180 KB raw / ~37 KB gzipped for ~147 items *including bodies*, so the entire dataset can ship to the client — no two-tier index/body split is needed yet. Because it's committed (not just a `dist/` artifact), it doubles as a linkable open-source database.
  - Generated by `scripts/build-db.mjs` (plain Node + the `yaml` dep) which walks `src/content/<category>/<locale>/*.md`, parses frontmatter, applies the same defaults as the Zod schema, and writes deterministic (sorted, no timestamps) JSON so re-running only changes the file when content actually changed. **The schema defaults are duplicated by hand in that script** — `content.config.ts` imports astro-only modules and can't be `require`d from plain Node, so if you add a defaulted field to the schema, add it to `BASE_DEFAULTS`/`CATEGORY_DEFAULTS` there too or the JSON will omit it.
  - Wired as **`prebuild` and `predev`** in `package.json` (plus a manual `npm run build:db`), so `npm run build`/`npm run dev` always regenerate it first and the committed copy can't silently drift from the markdown. The build then copies it into `dist/db/content.json` like any other `public/` file.
- **View Transitions + prefetch** — `<ClientRouter/>` (from `astro:transitions`) is in the `<head>` of *both* layouts, and `astro.config.mjs` sets `prefetch: { prefetchAll: true, defaultStrategy: 'hover' }`. Together these give SPA-style, no-full-reload navigation (links prefetched on hover) over the still-fully-static build — every page remains real pre-rendered HTML for SEO/direct links. The per-item detail pages are unchanged; only navigation between them got smoother.
- **Slide-over detail panel** (`DetailPanel.astro`) — the payoff of the client dataset. On the map homepage (`/`) and the web grid (`/web/`), clicking a marker's "Részletek →" button or a grid card opens the item in an overlay rendered *entirely from the in-memory `content.json`* (no navigation), so the map/grid and its live filter/zoom state stay mounted behind it. The URL is updated (History API) to the item's real static page, so refresh/share still loads the pre-rendered detail page. See the component note below for the mechanics and the one browser-only caveat.

## Content model

`src/content.config.ts` defines a `base` Zod schema shared by every category, then extends it per category with category-specific fields via `collectionFor(category, extraFields)`.

### Base fields (every item, every category)

| Field | Type | Notes |
|---|---|---|
| `title` | string | |
| `locale` | `hu \| uk \| us` | Only `hu` has content. Locale is a data dimension, not Astro's built-in i18n routing — adding UK/US later is just adding content files with `locale: uk`/`us` and activating those routes, no schema change. |
| `topics` | array of `adhd \| autism` | min 1. Items can be tagged both. |
| `summary` | string | |
| `tags` | string array | free-form, default `[]` |
| `sourceUrl` | URL | canonical link to the actual thing (purchase page, show page, video, clinic homepage) |
| `coverImage` | string, optional | path under `/covers/`, e.g. `/covers/szetszortsag.jpg`. Self-hosted, not hotlinked — see curation-methodology.md |
| `ratings` | array of `{source, score, scale, url?, note?, checkedAt}` | generic — no aggregation/weighting logic exists yet. Empty array is fine and expected when no real rating exists for that content type (see below). |
| `availability` | `in-stock \| unavailable`, default `in-stock` | for retail-style items (books) going out of print/stock. Don't repurpose this for clinic intake status — use `priceNotes` free text instead (see diagnosis). |
| `entryType` | `full \| episode`, default `full` | `episode` = a one-off piece of content from a channel/show that isn't itself dedicated to ADHD/autism (e.g. one Partizán Podcast episode). `full` = the whole show/channel/book/clinic is the entity. |
| `parentTitle` | string, optional | required alongside `entryType: episode` — the name of the parent show/channel, rendered as a badge on both listing and detail pages. |
| `curatedBy` | `manual \| agent`, default `manual` | not yet used by any actual agent pipeline, but the field exists for when one lands. |
| `publishedDate` | string, optional | when the content was published/released — for content types that have one (books, articles, videos, podcasts…). Plain string (`"YYYY"`/`"YYYY-MM"`/`"YYYY-MM-DD"`, **quoted** in YAML) so partial dates render as-is. Place-based entries omit it; `movies`/`research` use their own numeric `year` instead. Renders generically as "Megjelenés"; feeds JSON-LD `datePublished` (falling back to `year`). |
| `lastReviewed` | date | when the entry's data was last **checked against its source and found still correct** — set it to the verification date even if nothing was edited. Not the file-creation date. |
| `lastEdited` | date | when the entry's content was last **actually changed**. Required on every entry (seeded from git last-commit date; author-maintained thereafter). Distinct from `lastReviewed` — a re-verified-but-unchanged entry gets a fresh `lastReviewed`, same `lastEdited`. Shown in the detail footer / panel meta; feeds JSON-LD `dateModified`. |
| `ageGroup` | `children \| adults \| both`, default `both` | moved here from `development`'s category-specific fields when the map-first homepage needed to filter by age across every pin-able category, not just one. Backfilled with real (non-default) values on schools (all → `children`), diagnosis, and equipment by reading each entry's own `summary`/`tags` — no new research needed, just inferring from what was already written. `communities` now carries real per-item values too (adult clubs/forums → `adults`, parent groups → `children`, umbrella orgs → `both`) since it was seeded with real content. |

### Category-specific fields

- **books**: `author`, `isbn?`
- **podcasts**: `creator`, `platform`, `episodeCount?`
- **videos**: `creator`, `platform`, `viewCount?`, `subscriberCount?` — YouTube has no public star ratings anymore, so view/subscriber counts are the honest substitute; they live here, not in `ratings`.
- **influencers**: `handle`, `platforms` (array, min 1)
- **articles**: `publication` (string, required — the outlet, e.g. Telex/HVG360/Qubit), `author?`, `publishedDate?` (an **optional quoted string** `"YYYY-MM-DD"`, *not* `z.coerce.date()` — kept a plain string so it renders as-is in both the detail page and `DetailPanel` without date-formatting special-casing; must be quoted in YAML or the `2026-02-15`-style value would parse to a `Date` and fail the `z.string()` check). Written journalism/explainers about ADHD/autism from reputable Hungarian outlets — text pieces, no geo, lives on the `/web/` grid. `sourceUrl` is the article link. Distinct from `videos`/`podcasts` (media) and from clinic content-marketing blogs (deliberately excluded — see curation-methodology.md). Labels: `publication` → "Kiadvány", `publishedDate` → "Megjelenés" (added to `FIELD_LABELS` in both `[slug].astro` and `DetailPanel.astro`); homepage-grid emoji 📰.
- **equipment**: `priceRange?`, `retailerUrls` (array, default `[]`), plus optional `city?`/`address?`/`lat?`/`lng?` — added for the map-first homepage; almost every retailer sourced this session is online-only (no visitable location), so only one entry (`calmer-fuldugo.md` / profi-fuldugó.hu, Esztergom) actually has these populated. Don't expect most equipment entries to ever have a location — that's the honest state of Hungarian sensory-equipment retail, not a gap to rush-fill.
- **food**: `kind` (`recipe \| product \| guide \| supplement`), `dietaryTags` (default `[]`) — recipes, eating-guidance articles, and evidence-based supplements about ADHD/autism nutrition. No geo fields; lives on the `/web/` grid, never the map. `guide` = dietitian/expert eating-guidance articles (selective eating, GFCF-diet honesty, general ADHD nutrition); `supplement` = evidence-based vitamins/supplements only (omega-3, vitamin D — see curation-methodology.md for the strict honesty bar this category follows). `recipe`/`product` are also available (`product` currently unused). `kind` values are mapped to Hungarian labels (Recept/Termék/Útmutató/Étrendkiegészítő) via a small `KIND_LABELS` map **duplicated in both** `[slug].astro` (the static detail page) and `DetailPanel.astro` (the slide-over rendered from `content.json`) — keep the two in sync. `kind` is the only category field whose *value* is display-translated; movies' `kind` still renders raw in both. `kind` has no schema default, so it needs no entry in `build-db.mjs`'s `CATEGORY_DEFAULTS`.
- **communities**: `providerType` (`group-therapy \| therapist \| institution \| online-group`), `city?`, `address?`, `lat?`, `lng?`, `contact?` — peer-support groups, self-advocacy associations, community centers, and online communities (forums/Facebook groups). `online-group` is the type for online-only communities (Facebook groups, `asperger.hu`'s forum); those have **no** `city`/`lat`/`lng` (which is why `city` is optional here, unlike diagnosis/schools where it's required) and only ever appear in the `/hu/communities/` listing, never on a map. Physical entries (an association's office, a community center) carry the same geo shape as diagnosis (`city`/`address?`/`lat?`/`lng?`) and do appear on the map. `lat`/`lng`/`address` were added to this schema when the category was first seeded with real content — the map-first homepage's `index.astro` had always *tried* to read `lat`/`lng` off communities, but the fields didn't exist until then, so no community could ever pin before. Facebook-group entries can't have their live member count fetch-verified (login wall), so member counts come from public search data and that limitation is stated in the entry body — see curation-methodology.md.
- **diagnosis**: `providerType` (`doctor \| clinic \| foundation \| hospital \| expert-committee`), `city`, `address?`, `lat?`, `lng?`, `contact?`, `priceNotes?`. `expert-committee` = the state **Szakértői Bizottságok** (SNI-assessment committees, ~1/county), added for GitHub issue #1.
- **schools**: same shape as diagnosis, `providerType` is `mainstream-integration \| special-needs-school \| kindergarten` instead.
- **funding / access** (`funding: state \| private \| mixed`, `referralRequired: boolean`, both optional): mixed into `diagnosis`/`schools`/`development` (alongside `locations`/`locationStatus`) for GitHub issue #1, powering the map's "közfinanszírozott" (public only) filter. `state` = TB/köz-financed, `private` = self-pay, `mixed` = both; `referralRequired` = beutaló needed. Set on every public entry, backfilled on existing entries from their own content (the `ageGroup`-backfill precedent); entries whose model couldn't be classified are left unset (not guessed). The **Pedagógiai Szakszolgálatok** live in `development` (korai fejlesztés) and share addresses/coords with their diagnosis `expert-committee` counterparts — same institution, two services (terc/ambulancia precedent).
- **development**: therapists/psychologists/centers doing ongoing developmental/therapeutic work (ABA, sensory integration, skills groups, gyógypedagógia) — distinct from `diagnosis` (getting diagnosed) and `communities` (peer support groups/institutions). Same geo shape as diagnosis/schools (`city`, `address?`, `lat?`, `lng?`, `contact?`, `priceNotes?`) plus `providerType` (`psychologist \| therapist \| developmental-pedagogue \| occupational-therapist \| center`) — most real entries are multi-practitioner centers (`center`), individual solo practitioners are the minority. `ageGroup` used to live here specifically; it's on `base` now (see above). The same institution can legitimately have one entry here and a separate one in `diagnosis` if it runs both services (e.g. Autizmus Alapítvány: `autizmus-alapitvany-ambulancia` in diagnosis, `terc` here) — that's not a duplicate, it's two different services at the same address.
- **movies**: `kind` (`documentary \| narrative`), `format?` (`film \| series` — **omitted means film**; set `series` for TV series), `director?`, `creator?` (series showrunner — `director` is film-oriented), `year?` (first-air year for series), `seasons?` (series only), `runtimeMinutes?` (films; omit for series), `streamingUrls` (array, default `[]`) — films, **series**, and documentaries about ADHD/autism, or with a well-known ADHD/autistic protagonist. Series were added on request ("films can contain series too"); they live in this same category rather than a new one, distinguished only by `format: series` — so a query for "all screen content" is just the movies collection. `format` is left off films (rather than defaulting to `film`) specifically so the 30+ existing film entries don't all sprout a redundant "Formátum: film" row on their detail pages; only series carry the field. Has no geo fields, unlike equipment; belongs on the `/web/` grid page, never the map. `availability` (a base field) is reused here for "watchable in Hungary right now" (`in-stock`) vs. "no legal/confirmed HU streaming option" (`unavailable`) — checked across **all** HU services via JustWatch (Netflix, Disney+, HBO Max, Amazon Prime, SkyShowtime, Apple TV…), not just one; see content-status.md and curation-methodology.md. The `[slug]` detail page renders `kind`/`format` values through a small `VALUE_LABELS` map (`documentary`→"dokumentumfilm", `narrative`→"játékfilm", `series`→"sorozat") so enum values show in Hungarian, not raw English — add to that map (in `[slug].astro`) if a new enum value needs a Hungarian display string. **`/web/` cards match the cover's real shape** (`POSTER_SHAPE` map in `WebExplorer.astro`, mirrored in `DetailPanel.astro`): **portrait** covers — `movies` (film posters) and `books` (~2:3) — and **square** covers — `podcasts` (1:1 art) — get a **horizontal card, cover left / text right** (`flex-row`, cover `w-24` with `aspect-[2/3]` or `aspect-square`, `object-cover`) so the artwork isn't cropped; everything else keeps the **vertical landscape top-thumbnail** (`h-32 object-cover`) because its covers really are landscape — `videos` are 16:9 YouTube thumbnails, `equipment` are wide product shots. The `DetailPanel` slide-over mirrors this: portrait/square covers shown **centred and uncropped** (`object-contain`, capped height), landscape ones full-width. The `[slug]` static detail page and the `/hu/<category>/` listing already place the cover left of the text (small width-constrained thumbnail, aspect preserved), so they need no per-shape logic. To change a category's treatment, edit the one `POSTER_SHAPE` map in each of the two components.

- **research**: `authors`, `venue` (journal/publisher/registry), `year`, `publicationType` (`journal-article \| review \| clinical-trial`), `language` (`hu \| en` — the *paper's* language; the `summary` is **always** written in Hungarian regardless), `doi?` (DOI, repository handle, or an `NCT…` trial id), `openAccess?`. Peer-reviewed papers/reviews and registered clinical trials on ADHD/autism. **Deliberately not Hungarian-only**: the curation goal (per the user) is the *best, most relevant* research — Hungarian-language work, Hungarian-authored/Hungarian-population studies, **and** hugely-relevant international landmarks — each with a Hungarian summary. `language` marks whether the source itself is HU or EN so that's transparent. Non-geo, like books/movies: it's a `WEB_CATEGORIES` member (shows on `/web/` and gets a homepage card, 🔬), never on the map. `sourceUrl` points at the DOI/registry/open-repository page. Enum values render in Hungarian via the shared `VALUE_LABELS` map (`review`→"áttekintő tanulmány", `clinical-trial`→"klinikai vizsgálat", `hu`→"magyar", `en`→"angol", `true`/`false`→"igen"/"nem" for `openAccess`).

`CATEGORIES` (exported from `content.config.ts`) is the single source of truth for which categories exist. Display metadata (label + emoji), field labels, enum-*value* labels, and browsing-surface grouping now live once in **`src/lib/categories.ts`** — `CATEGORY_META`/`CATEGORY_LABELS`/`FIELD_LABELS`/`VALUE_LABELS`/`WEB_CATEGORIES`/`MAP_PIN_CATEGORIES`/`LISTING_MAP_CATEGORIES` — and the page files plus `DetailPanel.astro` import from there instead of each keeping a copy. (They used to; `movies` got left out of the homepage copy and rendered a blank card, which is what motivated the consolidation.) Adding a category = add it to `collections` in `content.config.ts`; add a `CATEGORY_META` entry (with emoji) plus any new `FIELD_LABELS`/`VALUE_LABELS`, and — if web-grid-browsable — add it to `WEB_CATEGORIES`, all in `src/lib/categories.ts`; create `src/content/<category>/hu/`. The `/web/<category>` route, its tab bar, and the back-links all derive from `WEB_CATEGORIES`, so the only extra manual touch for a new web category is `WebExplorer.astro`'s `WebItem` category union (a TS literal type). (The old `TYPE_FILTERS`/`ALL_TYPES` chip lists are gone — the type filter became per-category permalink pages.) `build-db.mjs` discovers categories from the filesystem, so it needs a change only if the category adds a *defaulted* field (to `CATEGORY_DEFAULTS`). `articles` and `research` were added 2026-07-05.

### Multiple locations (mappable categories)

`diagnosis`, `schools`, `development`, `communities`, and `equipment` can carry **more than one physical site** without duplicating the whole entry (an early mistake modeled a single foundation as two files — see below). Two mixed-in fields (defined once as `multiLocation` in `content.config.ts`):

- **`locations`** — optional array of *additional* sites **beyond** the primary (top-level `city`/`address`/`lat`/`lng`). Each: `label?`, `city`, `address?`, `lat?`, `lng?`, `contact?`, `priceNotes?`, `note?`. The top-level fields stay the primary site, so the single-site majority needs **no migration** and the field simply doesn't appear on them.
- **`locationStatus`** — the research outcome, so multi-location coverage is auditable: `single-confirmed` (verified sole site), `multiple-confirmed` (extra sites found, in `locations`), `unsure` (couldn't determine). Absent = not yet researched. See [docs/multi-location-research.md](multi-location-research.md) for the per-entry log and method.

Both are optional with no schema default, so **`build-db.mjs` needs no change** — it spreads all frontmatter (`...data`) and `normalizeDates` recurses into the array. Neither is defaulted, so nothing goes in `BASE_DEFAULTS`/`CATEGORY_DEFAULTS`.

**Rendering derives one map pin per site** (primary + each `locations[]` entry that has `lat`/`lng`), all linking to the same detail page:
- `index.astro` (homepage) and `[category]/index.astro` (listing) flatten each entry to per-site pins. On the homepage each pin carries a **`key` of `href#siteIndex`** so `MapExplorer`'s `markersByKey` can add/remove individual pins on filter changes without collisions (keying by bare `href` would collide across an item's own sites). On the listing, the **Budapest vs. vidék split is per-site**, so a clinic with a Budapest branch and a countryside one pins on both maps.
- `[slug].astro` (detail) and `DetailPanel.astro` render a **"További helyszínek"** section listing the additional sites (+ the `locationStatus` badge, excluded from the generic field `dl`). The detail map (`ClinicMap`) shows *all* sites; its region is the tight Budapest box only when every site is in Budapest, else the national view so no pin is clipped. JSON-LD emits every site as a schema.org `Place` when there's more than one.

**The MARS lesson (data-modeling caution):** "MARS Alapítvány" turned out to be a Budapest advocacy foundation whose "diagnózisközpont kereső" is a *referral directory* of ~24 independent providers, not its own branches — its two former entries were mis-attributed and re-mapped. When adding locations, only count sites the **same institution actually operates**, never orgs it refers to or a same-named institution elsewhere. This is enforced in the research prompts and documented in `docs/curation-methodology.md`.

## Routing

- `src/pages/index.astro` — **the landing chooser**, `/`. Not under `[locale]` (Hungary-only; other countries get their own domains). Uses the standard `Layout` (header **and** footer), not the full-bleed `MapLayout`. Two selector groups of illustrated cards: (1) audience — "Mi jellemez a leginkább?" *Szülő* (→ `children`) vs *Felnőtt* (→ `adults`); (2) mode — Térkép/Web/Chat linking to `/map`, `/web/`, `/chat`. The audience pick is optional and non-destructive: a small script highlights the chosen card, remembers it in `localStorage` (`neuro-audience`), and appends **`?age=children|adults`** to the three mode links so the destination opens with its age filter preset (with JS off, the mode links still work param-less). Illustrations are inline SVG (no external assets, CSP-safe). This replaced the map, which moved to `/map`.
- `src/pages/map.astro` — **the map browsing surface**, `/map` (the old `/` content, unchanged). Pulls every geo-tagged item (one pin per site, see "Multiple locations") across `diagnosis`/`schools`/`development`/`communities`/`equipment` into `<MapExplorer>` + `<DetailPanel>` inside `<MapLayout>`. `MapExplorer` reads `?age=` on init and lets it win over saved filter state (the landing's audience choice).
- `src/pages/chat.astro` — **the conversational AI surface**, `/chat`, in `MapLayout` (full-bleed, header, no footer). A classic chat UI — thread with assistant/user bubbles, suggestion chips, an auto-growing composer (Enter to send, Shift+Enter for newline), a typing indicator. **Backed by a real Ollama Cloud assistant with tools** (see "Chat assistant" below): the page runs a client-side tool loop against `/api/chat`, and the model's tools (search/pull site content, produce links + resource cards) execute in the browser against the preloaded `content.json`. Model prose is rendered through a **sanitized markdown subset** (escape-first; only bold, internal/`http(s)` links, and bullet lists survive — no raw model HTML reaches `innerHTML`). This is the first feature that isn't pure static — it needs a server endpoint (the key can't ship to the browser), which is why the repo now has a `functions/` layer.
- `src/pages/web/[category].astro` — the **per-category grid**, `/web/<category>`, one **permalinked fixed list per web content type** (`books`, `podcasts`, `videos`, `equipment`, `movies`, `food`, `articles`, `research`). `getStaticPaths` iterates `WEB_CATEGORIES`. Replaced the old single combined `/web` grid with type-toggle chips — mixing e.g. movies and articles in one list wasn't useful, so each type is its own page reached by a **tab bar of navigation links** (rendered by `WebExplorer`), and only the **age toggle** still filters (within the one category). Same `MapLayout` + `<DetailPanel>` as the map. The tab bar and age toggle carry `?age=` (see below). `/web` (bare) → **redirect** to `/web/books` via `redirects` in `astro.config.mjs` (query strings aren't preserved across a static redirect, so the landing card and header link straight to `/web/books`). Adding a new web category: add it to `WEB_CATEGORIES` in `categories.ts` (the route, tab bar, and back-links all derive from it) and update `WebExplorer.astro`'s `WebItem` category union.
- `src/pages/rolunk.astro`, `adatvedelem.astro`, `gyik.astro` — About/Privacy/FAQ pages linked from `SiteHeader`'s nav, real Hungarian copy (not placeholder). `adatvedelem.astro` documents actual data practices honestly (no accounts/analytics/cookies; geolocation is client-side-only, never sent to a server) and marks the operator's legal identifiers (cégjegyzékszám, adószám, székhely) as **not yet published** rather than inventing them — "Armin & Felix Kft." doesn't turn up in the Hungarian company registry (Nemzeti Cégtár) under a search for those details, so don't treat that name as a verified registered entity without the user supplying real registration data.
- `src/pages/[locale]/index.astro` — the **original** category-grid homepage, still fully functional at `/hu/`, just no longer the default landing page. Kept intentionally, per the user's "keep the current front page for reference or later" instruction — don't delete or "clean up" this route without checking first.
- `src/pages/[locale]/[category]/index.astro` — listing. Topic filter (`Mind`/`ADHD`/`Autizmus`) is **client-side**: these pages are statically prerendered, so a `?topic=` query param can't change the server output (every request gets the same HTML) — this used to filter on `Astro.url.searchParams` server-side and was a silent no-op in production. Now the page ships every card and JS toggles a `hidden` class, reading the initial topic from the URL and keeping the URL in sync via `history.replaceState`. (The embedded map still shows all located items regardless of topic.) Renders `<ClinicMap>` above the list when the category is in `LISTING_MAP_CATEGORIES` (`diagnosis`, `schools`, `development`, `communities`) and at least one item has `lat`/`lng`. The "back" link at the top routes to `/web/<category>` (that type's own list) when the category is in `WEB_CATEGORIES` (`books`, `podcasts`, `videos`, `equipment`, `movies`, `food`, `articles`, `research`) and to `/map` otherwise — so content always links back to whichever browsing surface it's actually reachable from, not a hardcoded default.
- `src/pages/[locale]/[category]/[slug].astro` — detail page. `getStaticPaths` walks every category's collection and derives `{locale, category, slug}` from the content file's `id` (glob loader IDs are `<locale>/<filename-without-extension>`, so `slug = id.split('/').slice(1).join('/')`). Renders every extra (non-base) field generically via a label lookup map (`FIELD_LABELS`) — adding a new category-specific field to the schema means adding its Hungarian label there too, or it'll render with the raw key name. Same `WEB_CATEGORIES` back-link routing as the listing page above.

## Chat assistant (Ollama Cloud + client-side tools)

The `/chat` assistant is the **first and only server-backed feature** in an
otherwise fully static site. Design:

- **`functions/api/chat.ts`** — a Cloudflare **Pages Function** (net-new `functions/`
  layer; no Astro adapter, `output` stays `'static'`). It is a **thin, stateless
  proxy**: receives the full message history, prepends the system prompt, attaches the
  tool schemas, adds `Authorization: Bearer $OLLAMA_API_KEY`, and POSTs to
  `https://ollama.com/api/chat` (`stream:false`, model `OLLAMA_MODEL` ?? `gpt-oss:120b-cloud`),
  returning just the assistant message. It never runs the tool loop and never touches
  the content dataset. `OLLAMA_API_KEY` is a **real secret** (no `PUBLIC_` prefix) — it
  must be set in the Cloudflare Pages env (and a local `.env` for `wrangler pages dev`);
  a missing key returns a graceful 503 the UI shows as an error bubble.
- **Client-side tool loop** (`chat.astro` script): the page holds the conversation and
  loops (capped at `MAX_TOOL_ITERATIONS = 5`) — POST to `/api/chat`, and if the model
  returns `tool_calls`, execute each **in the browser**, append `{role:'tool', tool_name,
  content}`, and POST again; otherwise render the final text and stop.
- **`src/lib/chatTools.ts`** — the client-side tool executors + the coupling point with
  the function (only the tool **names** are shared; the schemas live in the function).
  `search_resources`/`get_resource` search the preloaded `content.json` (fetched +
  memoized exactly like `DetailPanel`, keyed by `href`; accent-folded scored match over
  title/tags/summary/city/body). `show_resource`/`link_to` return a `render` payload
  (`resourceCardHtml`/`linkChipHtml`, reusing `CATEGORY_META` for badge label+emoji) that
  the chat appends to the thread, plus a short confirmation string for the model. Because
  content search runs client-side, **the lookups never leave the browser** — only the
  conversation text is sent to Ollama (privacy: disclosed in `adatvedelem.astro`).
- Chat links point at the **real pre-rendered detail pages** (`href` from `content.json`),
  not the `DetailPanel` overlay — `DetailPanel` is not mounted on `/chat`.
- **Deferred (fast-follow, per Rooster NEURO-2):** a map-driving tool that presets the
  map filters (`neuro-map-filters` + `?age=`) and recenters location; and token streaming
  in the function (currently `stream:false`).

### SEO & discoverability

- **`site` in `astro.config.mjs`** comes from `process.env.SITE_URL` (fallback: the Cloudflare Pages default subdomain). It powers canonical/OG URLs, the sitemap, and robots — **set `SITE_URL` to the real production domain** in the Cloudflare Pages env or canonical/sitemap URLs point at the placeholder. See `docs/deployment.md`.
- **`@astrojs/sitemap`** integration emits `/sitemap-index.xml` + `/sitemap-0.xml` at build.
- **`src/pages/robots.txt.ts`** is a static endpoint (not a `public/` file) so the `Sitemap:` line derives from `site` and can't drift from the real domain.
- **JSON-LD structured data** — `[slug].astro` emits a `<script type="application/ld+json">` per item, with the `@type` mapped by category (`MedicalClinic`/`EducationalOrganization`/`Article`/`ScholarlyArticle`/`Book`/`Movie`/…) and only the fields we actually have (geo+`PostalAddress` for located entries, author/publisher/date for articles & research). Absolute URLs come from `site`.
- **`src/pages/404.astro`** — branded not-found page (static hosts serve it for any unmatched path) with links back to the map and the web grid.

### Accessibility

- **Filter state is exposed via `aria-pressed`.** The type/age filter chips (`MapExplorer`, `WebExplorer`) and the topic pills (`[category]/index.astro`) are toggle `<button>`s whose selected state was previously conveyed only by color; the filter scripts now keep `aria-pressed` in sync as they toggle, so screen readers announce what's active. (Set at runtime by the same JS that toggles the visual state.)
- **Reduced motion.** `global.css` has a `@media (prefers-reduced-motion: reduce)` block that near-zeroes transition/animation durations site-wide (View Transitions, the slide-over, chevrons), and the map-marker `BOUNCE` is separately gated in `MapExplorer` (the media query can't reach a JS-driven Google Maps animation).

### Why Google Maps, and what it costs

This was Leaflet + OpenStreetMap originally (free, no API key, no billing account, ever) — see `docs/gotchas.md` for Leaflet-specific traps that no longer apply now that it's gone. Switched to Google Maps JS API on request. Requires: a **Maps JavaScript API** key from a Google Cloud project with **billing enabled** (mandatory even to stay inside the free tier — no billing account = the API won't serve requests at all), and that key **restricted by HTTP referrer** to this site's domain(s) so it can't be lifted and abused elsewhere. Pricing as of this write-up: first 10,000 map loads/month free, then $7/1,000 loads — a low-traffic directory site should realistically stay at $0, but there's no "always free, zero account" tier for a real embedded map with custom markers (the genuinely-free options — the legacy `output=embed` iframe, or the Maps Demo Key — don't support custom markers/popups or aren't meant for production, respectively). The key lives in `.env` as `PUBLIC_GOOGLE_MAPS_API_KEY` (the `PUBLIC_` prefix is required for Astro/Vite to expose it to browser code — this is fine because a Maps JS key is meant to run client-side and is secured by domain restriction, not by being secret) and must also be set in Cloudflare Pages' project environment variables for production builds, not just locally — see `docs/deployment.md`.
